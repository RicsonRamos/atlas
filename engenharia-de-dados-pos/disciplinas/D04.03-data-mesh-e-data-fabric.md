# Disciplina 12

## Arquiteturas Modernas de Dados (Data Mesh e Data Fabric)

**Carga Horária:** 40 horas

## Ementa

Análise crítica e comparação epistemológica e tecnológica das arquiteturas modernas de processamento e gerenciamento de dados de larga escala. Limitações fundamentais das abordagens centralizadas tradicionais, como Data Warehouses e Data Lakes corporativos monolíticos. A arquitetura Data Mesh: descentralização sociotécnica orientada a domínios, dados como produtos (Data Products), infraestrutura de dados self-service como plataforma e governança federada computacional. A arquitetura Data Fabric: orquestração dinâmica e integração automatizada de dados baseada em metadados ativos (Active Metadata), grafos de conhecimento e inteligência artificial aplicada ao gerenciamento e descoberta de dados. Análise de interoperabilidade, padrões de comunicação e contratos de dados (Data Contracts). Padrões de design organizacional e engenharia de plataformas distribuídas de dados em escala de mercado.

## Objetivos

Ao final da disciplina o estudante deverá ser capaz de:
1. Analisar criticamente as limitações arquiteturais de plataformas centralizadas de dados sob a ótica de gargalos organizacionais e técnicos em escala corporativa.
2. Contrastar, sob a ótica de design, governança e processamento, as abordagens conceituais e tecnológicas do Data Mesh e Data Fabric.
3. Projetar e modelar um ecossistema de dados distribuído baseado nos quatro princípios fundamentais do Data Mesh, definindo as características arquiteturais e o ciclo de vida de um Produto de Dados (Data Product).
4. Desenvolver sistemas de governança de dados automatizados baseados em políticas descritas como código integradas a infraestruturas de dados autoatendidas.
5. Projetar a arquitetura técnica de um Data Fabric utilizando grafos de conhecimento e metadados ativos para automatizar pipelines de integração em ambientes multi-cloud.

## Competências

1. **Modelagem de Arquiteturas de Dados Descentralizadas**: Habilidade para conceber e projetar sistemas de dados complexos que distribuem a responsabilidade de processamento e propriedade para domínios de negócios.
2. **Definição de Contratos e Padrões de Dados**: Competência para modelar contratos de dados (Data Contracts) formais de modo a garantir interoperabilidade e confiabilidade em barramentos e malhas de dados distribuídos.
3. **Engenharia de Plataformas Autoatendidas (Self-Serve Platforms)**: Capacidade de projetar plataformas de infraestrutura como serviço (IaaS/PaaS) que suportem equipes de domínio no ciclo de vida de desenvolvimento de dados.
4. **Governança Federada Computacional**: Capacidade de automatizar regras globais de governança corporativa, privacidade de dados e conformidade regulatória através de código na infraestrutura distribuída.

## Habilidades

1. Mapear e estruturar produtos de dados, especificando suas portas de entrada, portas de saída, controle de acesso e SLOs associados.
2. Projetar esquemas de validação sintática e semântica de dados distribuídos e monitoramento automático via Data Contracts.
3. Implementar grafos de conhecimento que indexam relacionamentos semânticos entre dados de múltiplos sistemas heterogêneos.
4. Construir automações de provisionamento de recursos de infraestrutura analítica utilizando práticas de infraestrutura como código (IaC).

## Conteúdo programático

**Unidade I – Crise Analítica Centralizada e Evolução das Arquiteturas**
1. Evolução histórica: Data Warehouses tradicionais, Data Lakes corporativos e Lakehouses analíticos.
2. Gargalos operacionais, organizacionais e de engenharia decorrentes de equipes e repositórios centralizados.
3. A transição sociotécnica em engenharia de sistemas de dados: descentralização, autonomia de domínios e redução de carga cognitiva.
4. Comparação conceitual entre paradigmas de dados distribuídos e sistemas centralizados.

**Unidade II – Data Mesh: Os Quatro Pilares Conceituais**
1. Arquitetura descentralizada orientada a domínios (Domain-driven data ownership).
2. Dado como Produto (Data as a Product): anatomia de um produto de dados, interfaces de entrada e saída (ports), usabilidade, confiabilidade e ciclo de vida.
3. Plataforma de infraestrutura de dados autoatendida (Self-serve data platform): plano de controle, plano de dados e planos de desenvolvimento (developer utilities).
4. Governança federada computacional (Federated computational governance): conformidade descrita como código, interoperabilidade global, padrões de comunicação e segurança policêntrica.

**Unidade III – Data Fabric: Arquitetura Inteligente Orientada a Metadados**
1. Conceito e fundações do Data Fabric: orquestração inteligente de dados e integração automatizada.
2. Metadados Ativos (Active Metadata) e Grafos de Conhecimento (Knowledge Graphs) para representação semântica corporativa.
3. Automação cognitiva de pipelines: descoberta automatizada de esquemas, mapeamento dinâmico e auto-otimização de consultas distribuídas.
4. O papel da inteligência artificial e aprendizado de máquina na orquestração inteligente de barramentos de dados corporativos.

**Unidade IV – Contratos de Dados (Data Contracts) e Interoperabilidade**
1. Contratos de dados (Data Contracts): definição de acordos estruturais, semânticos, SLAs de qualidade de dados e políticas de segurança.
2. Implementação prática de contratos de dados em pipelines de produção (JSON Schema, Avro, Protobuf).
3. Governança e evolução de esquemas (Schema Registry) e compatibilidade retroativa e futura.
4. Padrões de acesso distribuídos de dados: REST, gRPC, compartilhamento direto (SQL, Delta Sharing) e mensageria distribuída.

**Unidade V – Design Organizacional e Casos de Uso Avançados**
1. Estruturas organizacionais baseadas no Data Mesh: times de domínio, times de plataforma e times de governança federada.
2. Padrões de comunicação organizacional orientados a topologias de times (Team Topologies) aplicado a dados.
3. Análise crítica de estudos de caso: implementação de Data Mesh e Data Fabric em bancos, fintechs, e-commerce global e grandes corporações multinacionais.
4. Avaliação de custos, desafios de migração e trade-offs operacionais reais na transição para malhas e tecidos de dados.

## Laboratórios

1. **Laboratório 1: Definição e Validação de Contratos de Dados (Data Contracts)**: Projetar e implementar um contrato de dados usando especificação YAML/JSON Schema que define regras estruturais, semânticas e SLOs de um Produto de Dados. Desenvolver um pipeline de ingestão em Python/PySpark que valide os eventos de entrada contra o contrato, direcionando mensagens inválidas para uma Dead Letter Queue (DLQ).
2. **Laboratório 2: Provisionamento Automatizado de Infraestrutura Self-Serve**: Criar uma simulação de portal de autoatendimento onde equipes de domínio solicitam a criação de novos Produtos de Dados. O aluno escreverá scripts em Terraform e scripts de automação que, ao receberem uma requisição parametrizada, provisionam dinamicamente tópicos do Kafka, bancos de dados analíticos isolados no PostgreSQL ou diretórios analíticos (MinIO/S3) estruturados com controles de acesso RBAC.
3. **Laboratório 3: Criação de um Grafo de Conhecimento para Data Fabric**: Desenvolver um protótipo em Neo4j para mapear metadados corporativos. O estudante deve construir consultas em Cypher que identifiquem relações de dependência, tabelas órfãs, acessos a dados sensíveis por diferentes usuários e de que maneira os metadados técnicos de diferentes sistemas (banco relacional, data lake, relatório de BI) se integram semanticamente na organização.

## Estado da arte e debates em aberto

1. **Data Mesh vs. Data Fabric: Convergência ou Exclusão Mútua?**: Debate conceitual e prático sobre se as abordagens são opostas ou complementares. É viável implementar uma malha de dados distribuída orientada a domínios (Data Mesh) que se beneficie de um tecido de metadados ativos (Data Fabric) para automatizar a integração e descoberta global?
2. **A Sobrecarga Cognitiva das Equipes de Domínio**: Avaliação crítica sobre se a descentralização de dados impõe às equipes de negócios uma complexidade excessiva de engenharia de software e infraestrutura de dados, gerando ineficiências ou novos silos tecnológicos.
3. **Governança Global de Dados Sem Centralização**: O desafio de manter padrões mínimos de qualidade, conformidade regulatória (LGPD) e segurança da informação sem recorrer a comitês centrais lentos que bloqueiam a agilidade do desenvolvimento de produtos de dados pelos domínios.

## Avaliação

- **Projeto de Design de Produto de Dados**: Implementação de um protótipo de Produto de Dados contendo validação de contratos, infraestrutura autoatendida via Docker/Terraform e governança computacional descrita como código (50%)
- **Relatório Arquitetural Comparativo**: Elaboração de projeto detalhado de engenharia de dados propondo a adoção comparada de Data Mesh ou Data Fabric para uma grande corporação multinacional fictícia (30%)
- **Defesa de Arquitetura**: Apresentação oral e debate científico das decisões de descentralização e governança tomadas no projeto (20%)
